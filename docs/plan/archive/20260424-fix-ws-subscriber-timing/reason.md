# Reason — WS 구독 타이밍 버그 수정

## 왜 빨리 고쳐야 했나

MVP가 "테스트 11/11 통과"라고 보고됐지만, 실제로 사용자가 UI를 돌리자 "spawn도 안 되고 프론트도 삑나"라는 피드백이 왔다. 테스트는 "내가 호출하는 API 순서"를 검증했지, **UI의 자연스러운 사용 흐름**(WS open → spawn)을 재현하지 않았다. 이게 가장 아픈 교훈.

Axis 2(시각화)로 넘어가는 시점에 이 버그가 남아 있으면:
- UI에서 아무 데이터도 안 오는 상태에서 "도트 그래픽이 움직이게 만들기"는 근본적으로 불가능.
- 데이터가 오지 않는 상태를 "렌더러 버그"로 오진하면 시간을 크게 태움.

그래서 Axis 2 착수 전에 이 하나만 먼저 고정하는 계획을 분리했다.

## 왜 `SessionManager`를 쪼개는 방향인가

대안 비교:

### A. `subscribe`가 없는 세션을 새로 만들어 버린다 (기각)

`subscribe(repoId)`에서 세션이 없으면 빈 플레이스홀더 세션을 만들어 두고 나중에 register에서 handle을 채우는 방식. 
- 단점: 세션 = 핸들 보유자 라는 의미가 흐려진다. `has(repoId)`가 true인데 실제로 spawn은 아직 없는 유령 상태가 생긴다. 상위 로직이 복잡해진다.
- 결론: 기각.

### B. spawn 이벤트를 일정 시간 버퍼링한다 (기각)

register 직후 N초간 이벤트를 지연시켜, 사이에 붙는 WS가 놓친 이벤트를 받게 한다.
- 단점: 타이밍 힌트에 의존하는 불안정 설계. 숫자 튜닝 지옥.
- 결론: 기각.

### C. 구독자 맵과 세션 맵을 분리한다 (채택)

- 장점: 두 라이프사이클을 솔직히 분리. 순서 독립성이 생김. 수정 범위도 한 파일에 한정.
- 단점: 상태가 두 개의 맵에 나뉜다. 다만 이 둘은 의미가 확연히 달라 오히려 섞여 있는 지금이 잘못된 모델이었다.
- 채택.

## `lastEventByType` 리플레이를 유지한 이유

기존 구현도 마지막 이벤트를 타입별로 1건씩 기억하고 있었다. 이 기능 자체는 useful — 늦게 붙은 UI가 현재 status를 즉시 반영하려면 필요. 그래서 버그 수정과 별개로 유지했다. 다만 "세션 영속화"를 논의할 시점에 모델 재검토 예정 (예: 이벤트 로그를 SQLite/JSONL로 내려 쓰는 옵션).

## 디버깅 문서를 같이 쓴 이유

"테스트 통과 = 프로덕션 OK 아니다"라는 교훈을 다음 반복으로 전달하려면, 계층별로 실제 흐름을 재현할 수 있는 probe를 **상설 도구**로 남겨야 한다. 이번 버그를 잡은 방식이 그 doc의 초판 내용이 된다.

## 기술적 재발 방지

- 통합 테스트를 **UI 사용 흐름 순서**로 하나 더 작성하는 것을 고려. 이번엔 WS를 Node `WebSocket` 클래스로 서버에 연결해 HTTP spawn을 병행 검증하는 테스트가 필요. 현재는 session.ts의 단위 테스트로 순서 독립성을 고정했지만, 실제 HTTP/WS 레이어까지 포괄하는 테스트가 있으면 더 강력하다. → 후속 계획 아이디어.
- `bun test`는 빠르지만 분산 시스템의 타이밍 버그를 전부 잡지는 못함. Axis 2 들어갈 때 그래픽 테스트를 어떻게 조직할지 별도 고민이 필요.
