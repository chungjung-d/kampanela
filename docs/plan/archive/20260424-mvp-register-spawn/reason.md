# Reason — MVP: 레포 등록 + 단일 Claude Code spawn

## 왜 이 MVP 범위인가

- kampanela의 GOAL은 3축(오케스트레이션/시각화/공유 브레인)이지만, Axis 2·3는 모두 "레포에 어떤 에이전트가 돌아가고 있다"는 전제 위에 성립한다.
- Axis 1의 가장 작은 회로를 먼저 확정하면 이후 반복이 안정된 기반 위에서 이뤄진다.
- 반대로 시각화부터 시작하면 표시할 데이터가 가짜여서 스키마 결정이 섣부를 수밖에 없다.

## 대안 비교

### A. UI 먼저 — `~/.claude/teams/` 더미 데이터 시각화

- 장점: 사용자에게 보이는 결과물이 빠르게 나옴
- 단점: 실제 이벤트 포맷이 미확정 → 다시 뜯을 확률 높음
- 결론: **기각**. 허상의 스키마 위에 UI를 쌓으면 재작업 비용이 크다.

### B. Shared Brain 먼저 — 심볼링크 온보딩 스크립트

- 장점: 오케스트레이션 없이도 바로 쓸모 있음
- 단점: 연결할 "에이전트"가 없으면 효용 측정 불가. 심볼링크 자체는 일회성 스크립트라 후속 확장 빈약
- 결론: **기각 (후순위)**. Axis 3는 Axis 1이 살아 있을 때 가치를 측정할 수 있다.

### C. 오케스트레이션 회로 먼저 (채택)

- 장점: 실제 이벤트를 관찰하며 스키마를 검증 → Axis 2/3의 설계 근거가 생김
- 단점: 초기에 사용자 가치는 적다 (로그 뷰 수준)
- 채택 이유: 이후 계획의 **토대를 제공**한다. 가치 있는 결과물은 다음 두 계획(PixiJS 시각화, Brain 링크)에서 빠르게 만들 수 있다.

## 기술 선택 근거

- **Bun workspaces**: 초기 개발 속도와 단일 런타임(서버/CLI/빌드)을 우선. Node+npm보다 설치/실행이 한 덩어리로 빠르다.
- **server가 core를 감싸는 구조**: core를 순수 라이브러리로 두면 CLI / 테스트 / 다른 프론트엔드에서 재사용 가능.
- **PixiJS를 MVP에서 제외**: 도트 그래픽 애니메이션을 붙이면 프로세스 이벤트 포맷 결정이 굳지 않은 상태에서 렌더링 코드까지 묶이게 된다. 먼저 데이터부터 안정화한다.
- **registry 파일을 홈 디렉토리에 둠** (`~/.kampanela/repos.json`): 여러 사용자·여러 클론에서 단일 뷰를 갖게 하기 위함. 레포 내 설정은 공용 저장 비적합.

## 의존성 / 위험 분석

- **외부 의존**: `claude` CLI가 설치되어 있어야 한다 (사용자 환경 전제). 바이너리 부재 감지 시 명확한 에러.
- **위험**: `claude` CLI의 프로그래매틱 호출 인자(헤드리스 모드/프롬프트 전달 방식)가 본 설계와 맞지 않을 가능성. → 착수 전 spike 필수.
- **위험**: Bun의 `child_process` 계열 API 성숙도. 장기 실행 프로세스의 stdout 스트리밍이 기대대로 동작하는지 초기 검증 필요.
